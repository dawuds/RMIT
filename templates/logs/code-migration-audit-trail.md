## 1. Purpose and Scope

### 1.1 Purpose

This document serves as the formal template and governance framework for the **Code Migration Audit Trail**, a mandatory compliance artifact under Bank Negara Malaysia's (BNM) **Risk Management in Technology (RMiT) Policy Document**, specifically **Clause 10.13**. The Code Migration Audit Trail provides a comprehensive, immutable, and continuously maintained log of all code and configuration migrations executed across the technology environment of [Organization Name], spanning development, staging, user acceptance testing (UAT), and production environments.

The audit trail supports the institution's obligations to maintain audit accountability, enforce change governance, and demonstrate regulatory compliance to BNM and internal audit functions.

### 1.2 Scope

*Define the systems, environments, and business units covered by this audit trail. Include all technology stacks subject to RMiT obligations.*

This audit trail applies to:

- All application code, scripts, configuration files, and infrastructure-as-code (IaC) artifacts promoted between environments
- All technology systems owned or operated by [Organization Name], including core banking, digital channels, middleware, and supporting infrastructure
- Migration events initiated by internal staff, vendors, managed service providers, and automated deployment pipelines
- The following environments: Development → UAT → Staging → Production, and any lateral or emergency migrations thereto
- All business units within scope of [Organization Name]'s Technology Risk Management Framework

**Out of Scope:** [List any explicitly excluded systems, environments, or artifact types, e.g., read-only configuration exports, sandbox environments not connected to production data.]

## 3. Timestamp and Event ID

*This section defines the mandatory temporal and identity metadata captured for every migration event. Each record in the audit trail must be uniquely identifiable and time-anchored to an authoritative source.*

### 3.1 Event Timestamp

All migration events shall be recorded with a timestamp sourced from a synchronised, authoritative time server (NTP). Timestamps must be recorded in **ISO 8601 format (UTC+8 / MYT)**.

| Field | Format | Example | Required |
|---|---|---|---|
| **Event Timestamp** | `YYYY-MM-DDTHH:MM:SS+08:00` | `2026-03-05T14:32:11+08:00` | Mandatory |
| **Initiation Timestamp** | `YYYY-MM-DDTHH:MM:SS+08:00` | `2026-03-05T14:30:00+08:00` | Mandatory |
| **Completion Timestamp** | `YYYY-MM-DDTHH:MM:SS+08:00` | `2026-03-05T14:35:47+08:00` | Mandatory |
| **Time Zone** | MYT (UTC+8) | `MYT` | Mandatory |

### 3.2 Event ID

Each migration event shall be assigned a globally unique Event ID generated by the deployment or change management system. Event IDs must be immutable once assigned.

| Field | Format | Example | Required |
|---|---|---|---|
| **Event ID** | `MIG-YYYYMMDD-NNNNNN` | `MIG-20260305-000142` | Mandatory |
| **Change Request ID** | As per ITSM system | `CHG-2026-04821` | Mandatory |
| **Pipeline Run ID** | CI/CD system reference | `RUN-GH-98732` | If applicable |
| **Correlation ID** | Cross-system traceability token | `[UUID v4]` | Recommended |

## 4. Event Type and Severity

*Classify each migration event by type and assign a severity rating to enable risk-based monitoring and escalation. Event type and severity classifications must align with [Organization Name]'s Technology Risk Management Framework.*

### 4.1 Event Type Classification

| Event Type Code | Description | Example |
|---|---|---|
| `MTYPE-01` | Scheduled Production Deployment | Sprint release to production via approved CAB |
| `MTYPE-02` | Emergency / Hotfix Deployment | Critical bug fix outside standard release cycle |
| `MTYPE-03` | Rollback / Reversion | Reverting to previous stable version post-incident |
| `MTYPE-04` | Configuration Change | Environment variable, feature flag, or infrastructure config update |
| `MTYPE-05` | Database Schema Migration | DDL/DML changes applied to production database |
| `MTYPE-06` | Third-Party / Vendor Patch | External software update or security patch deployment |
| `MTYPE-07` | Infrastructure-as-Code Deployment | Terraform, Ansible, or equivalent IaC execution |
| `MTYPE-08` | Certificate / Key Rotation | TLS, API key, or cryptographic credential replacement |

### 4.2 Severity Rating

| Severity Level | Code | Definition | Escalation Requirement |
|---|---|---|---|
| **Critical** | `SEV-1` | Migration affecting core banking, payment systems, or data of >10,000 customers | Immediate notification to CISO and Head of Internal Audit |
| **High** | `SEV-2` | Migration affecting customer-facing digital channels or regulatory reporting systems | Notification to IT Risk within 4 hours |
| **Medium** | `SEV-3` | Migration affecting internal business systems or batch processing | Logged and reviewed at next daily standup |
| **Low** | `SEV-4` | Non-production migration or minor configuration update | Standard log capture; reviewed weekly |
| **Informational** | `SEV-5` | Automated, pre-approved, low-risk pipeline actions | Captured in audit log; no escalation required |

### 4.3 Outcome Status

| Status Code | Description |
|---|---|
| `SUCCESS` | Migration completed without errors; post-deployment validation passed |
| `PARTIAL` | Migration completed with warnings; partial rollback may have occurred |
| `FAILED` | Migration failed; full rollback executed |
| `ROLLED-BACK` | Migration was manually or automatically reverted post-deployment |
| `PENDING-VALIDATION` | Migration complete; awaiting post-deployment test sign-off |

| [Organization Name] Change Management Policy | Internal change advisory board and approval workflows |

### 7.2 Log Collection Methodology

*Describe the technical mechanisms by which audit log data is collected and centralised.*

- **Automated Log Ingestion:** All migration events are automatically captured by [SIEM / Log Management Platform, e.g., Splunk, Wazuh] via integration with the CI/CD pipeline and ITSM system.
- **Log Immutability:** Logs are written to a write-once, read-many (WORM) storage target to prevent tampering. Log integrity is verified using cryptographic hash chaining.
- **Centralised Repository:** All audit trail records are stored in [Centralised Log Repository Name/URL].
- **Real-Time Alerting:** Automated alerts are triggered for `SEV-1` and `SEV-2` events, and for any migration executed outside of approved change windows.

### 7.3 Validation Procedures

| Step | Description | Performed By | Frequency |
|---|---|---|---|
| Completeness Check | Verify all expected migration events are present in the audit log | Automated (Pipeline) | Per event |
| Hash Verification | Validate artifact SHA-256 hash matches the source repository | Automated (Pipeline) | Per event |
| Authorization Cross-Check | Confirm every production migration is linked to an approved Change Request | IT Risk / Internal Audit | Daily |
| Anomaly Detection | Identify unauthorized, off-hours, or failed migrations | SIEM Automated Alert | Continuous |
| Quarterly Log Review | Full review of migration audit trail for completeness and exceptions | Head of Internal Audit | Quarterly |

### 7.4 Retention Policy

| Log Category | Retention Period | Storage Tier | Legal Hold Provision |
|---|---|---|---|
| Production Migration Logs | 7 years | Hot (0–1 yr), Cold Archive (1–7 yr) | Yes |
| UAT / Staging Migration Logs | 3 years | Hot (0–6 mo), Cold Archive (6 mo–3 yr) | No |
| Development Migration Logs | 1 year | Hot | No |
| Failed / Rolled-Back Events | 7 years | Hot (0–1 yr), Cold Archive (1–7 yr) | Yes |
